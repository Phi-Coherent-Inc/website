The φCrypt suite, computed live in your browser via WebAssembly — no server, no pre-baked numbers. The suite is built around one design decision: the signature, the proof system, and the recursion all share a single prime field, so a signature's verification arithmetic is native to the circuits that prove statements about it. That is what the first tab shows — a mechanism demonstration of a zero-knowledge proof system carrying signature-verification work, verified live on this page and scoped precisely inside the tab. Around that core sit two independent hardness assumptions: hash-based (φHash: φSign, φCipher) and lattice-based (Module-LWE/SIS: φKEM, φDSA). The independence is at the assumption level, not the implementation — φHash is a shared component across the hash leg and φKEM's instantiation, not a firewall; φDSA alone has zero φHash dependency. The Security model tab carries the exact dependency map, the measured costs, and every caveat.

Pre-production. These primitives already run in shipped internal infrastructure (phi-sqlvfs, the mobile wallet's JSI bridges), and the suite is under external cryptanalysis review that has not yet reported. Until it does, treat every number on this page as this team's own estimate, not an audited guarantee — see Related packages for exactly what depends on what today.

one shared field · signature × proof system

A signature verification, verified in proof

φDSA lives on the same prime field as the suite's proof system and recursion (Q=1342832641), so signature-verification arithmetic runs natively inside the circuits — nothing is emulated in a foreign field. Below, your browser verifies real proofs produced by that machinery: the heavy legs of a signature verification — the lattice relation A·z − c·t = w checked pointwise across all 256 evaluation points, and the response-norm bound — each carried by a recursive proof whose every input cell is pinned to the public signature. Flip a byte and watch it reject. This is a mechanism demonstration: the callout below states the current capacity limit of the algebraic hash it relies on, and why that limit makes the demo illustrative rather than security-bearing today.
Exactly what is and is not proven here. First, and most important: the proof does not verify the production signature — it verifies a different one. Production φDSA runs SHAKE-256 for its entire Fiat–Shamir transcript, and SHAKE is not arithmetizable at any size, so the demo signs a transcript variant (φDSA-algfs) whose challenge is derived with a field-native algebraic hash instead. That hash is named and specified: a Poseidon-class x⁷ sponge, state width 3 (rate 2, capacity 1), a 3×3 Cauchy MDS layer, and 52 all-full rounds. The round count is an algebraic-degree figure (⌈128/log₂7⌉ ≈ 46 plus a statistical margin, addressing Gröbner-basis and interpolation attacks) and says nothing about generic attacks. With a single capacity element over a ~2³⁰ field, generic security is capacity-limited: roughly 2¹⁵ collision and 2³⁰ preimage work, regardless of round count, and the challenge this leg derives is a single field element (~2³⁰ challenge space). A parameter revision that derives the sponge's capacity and output width from the 128-bit target (a wider state) is in progress. Until it lands, what runs below is a mechanism demonstration of proof-carried verification, not a security-bearing signature verification. The hash is experimental: the arithmetization-friendly hash family (Poseidon, Rescue, Griffin, Anemoi) has a genuinely active cryptanalytic record, and this instance has had no external review. It verifies the algebraic-FS signature, not the SHAKE one a production φDSA holder would issue. The demo variant is also hint-free (the commitment w travels uncompressed), placing its forgery bound in the norm-below-q SIS regime the parameter derivation certifies. Second: the lattice relation and the norm bound are proof-carried; the challenge equality and the NTT consistency of the public signature are recomputed natively by the verifier from public data, which is sound precisely because that data is public. Third: this statement attests a public signature's verification; it does not hide the signature. We have measured exactly what a hidden version would take: the relation and norm proofs run in zero-knowledge today, and the check tying the two together survives hiding, but the fingerprint step that binds the signature to the message would then have to run inside the proof at roughly 264 times the zero-knowledge size budget — a different proof structure we have not built. So the hidden-authorization statement that ships today is the hash-leg one-time-signature proof, verified beside it below. Fourth: the proofs were generated natively (the prover is multi-threaded; this host cannot serve the COOP/COEP headers browser threads require). Verification runs live here in the thread-free verifier module.

Flip one character above to see the avalanche — measured side by side against SHA3-256, the NIST reference (its export here is validated against the official test vectors). Identical-looking diffusion beside the standard is the meaningful statement; an avalanche readout alone is not, since every hash, including broken ones, flips about half its bits.
Beyond avalanche. A single flipped bit changing ~50% of the output is necessary but not sufficient — plenty of broken hashes pass that check too. The test suite goes further: the Strict Avalanche Criterion (mean ≈0.5, worst single output-bit deviation <0.15 — no dead or stuck bit) and the Bit Independence Criterion (output bits stay near-uncorrelated under an input flip, max |correlation| <0.25). The nonlinear core (the H4 S-box shared with φCipher) is checked directly: differential uniformity 4 and linear bias 32, the same AES-grade optimum as the GF(2⁸) inverse — see tests/test_adversarial_hash.cpp.
phi_cipher · AEAD · φ-ROMix v2

φCipher

GF(2⁸) wide-trail SPN, encrypt-then-MAC. The passphrase is not hashed directly — a bare hash, however good, is GPU/ASIC-parallelizable against a human-chosen passphrase. The key comes from φ-ROMix v2 (default cost here): a 7.5 MB fill buffer, 121,393 randomly-addressed mixing rounds, φHash only at the final extraction step — a real, non-trivial memory cost, though modest next to Argon2id's typical recommended range (64 MB-1 GB); ≈46 ms per derivation in this single-threaded WASM build. The fill/mix core is φ-ChaCha14: standard, unmodified ChaCha quarter-round rotations (Fibonacci rotation amounts were tried and rejected — they halved AVX2 throughput with no security gain), but a non-standard round count (14, vs. IETF's 8/12/20) and φ-Weyl state constants in place of ChaCha's "expand 32-byte k". φKEM — key encapsulation (two-party)

Alice publishes a key. Bob encapsulates a fresh shared secret to it — and, separately, encrypts a message directly to it (φKEM + φHash-derived DEM key + φCipher AEAD). Alice recovers both. All keygen/encaps/decaps calls are millisecond-scale.
phi::sign · stateful hypertree + FORS (SPHINCS+-style)

φSign — hash-based signature

Hypertree of L(2)=3 layers × height L(4)=7, WOTS width 8 — 8 values per digit, i.e. 3 bits, distinct from LMS notation where w counts bits per digit (revised down from 13 after the width sweep measured 13 as strictly costlier for the same security — see the Security model tab), FORS F(7)=13 trees. Security rests only on φHash (second-)preimage / target-collision resistance — a position-tweaked reduction in the XMSS/SPHINCS+ style, its own construction rather than an SP 800-208 parameter set; no elliptic curves, no algebraic hardness. Genuinely slow here: this build is single-threaded (GitHub Pages can't serve the COOP/COEP headers multi-threaded WASM needs), so the hypertree's one-time-key fan-out runs serially — keygen ≈13s, sign ≈40s. A Service-Worker shim to inject those headers client-side was tried and reverted: Chromium doesn't reliably route Emscripten's own pthread-worker bootstrap fetch through it, so the module fails to load ("SharedArrayBuffer is not defined") rather than just staying slow — see phi-crypt-cpp/web/CMakeLists.txt for the traced failure. Both slow steps run in a Worker so the page stays responsive; verify is fast (≈0.16s). Native single-core, the same operations measure keygen ≈1.4s, sign ≈4.3s, verify ≈0.18s at this tier — the gap between those numbers and the browser ones is the single-threaded WASM tax; the gap between those numbers and SHAKE-based schemes is φHash's own cost, measured plainly in the Security model tab. On state: the identity here is ephemeral by design — the demo module never serializes secret material out of WASM memory, so a fresh keypair is built per page load and a one-time index can never be reused across reloads. Durable signers use the library's reserve-then-sign hook: the counter is persisted before each signature exists.

Independence, precisely. "Independent legs" is true of the hardness assumptions (hash vs. Module-LWE) and false of the implementation. φSign's security is entirely φHash. φCipher's key, above, no longer traces to a bare φHash call — but the memory-hard KDF's own last step is still one φHash call. φKEM here is instantiated with φHash as its extendable-output function (the suite's own choice — the KEM core is hash-agnostic behind an injected XOF, and an in-suite SHAKE-256/FIPS-202 instantiation, validated against the NIST test vectors, works identically against the same Module-LWE assumption), so its key-establishment path also depends on φHash behaving as a random oracle, not only on Module-LWE — a stronger ask than "produces good avalanche," and the gap between those two turned out to matter (see below). The one primitive on this page with zero φHash dependency, assumption or implementation, is φDSA — it uses SHAKE-256 throughout. Read plainly: φHash is a shared component with suite-wide blast radius, not a firewall between the two legs.
The φHash-XOF finding, the fix, and the process gap behind it. φHash pegs ~6% of ExpandMask's coefficients at the boundary vs. ~0.02% expected. The cause is not a routing choice, it is a fact about φHash: past its 196-byte fully-mixed block, φHash's "chain extension" step applies only the linear Triveni lift/unlift — no S-box, no phyllotaxis mixing, no round constants. The explanation needs no framework: a linear map iterated on itself cannot add entropy, so output past 196 bytes is an affine function of internal state. That is worse than "biased" — it is potentially state-recovering, and the frozen byte we measured (constant across 4,000 seeds at the old chandas=8 request size) is a symptom, not a bound on what is extractable. Fix: the cryptographic XOF is now hard-capped at chandas=3 (123 bytes), the largest request that provably never reaches the extension path, enforced by a compile-time assertion so a future caller cannot silently exceed it. Call-site audit: every secret-material caller already stays below the boundary — the KEM XOF at ≤123 bytes, φSign at ≤76, φDSA on SHAKE-256 — so no key-bearing path ever ran through the linear extension. The one caller that does request past 196 bytes is phi-currency's proof-of-work grinding, which is public and keyless (no secret to recover), so the linear extension there is not a cryptographic surface; it is not "out of scope," it is out-of-attack-surface, and the extension path stays reachable only from that non-secret context. The honest finding is the process one: a frozen output byte is exactly what a statistical battery catches instantly, and φHash had none gating its releases. It does now — a monobit/frequency/frozen-position battery over many seeds (tests/test_hash_battery.cpp) that fails the build on a stuck byte or an out-of-band frequency deviation. Re-measured clean at chandas=3 (saturation back to the expected 6.23%), KAT anchors re-pinned, decode-failure bound δ=2⁻²⁸⁴·⁷, full suites green.
Reading the hash-based rows, and why they are not birthday-limited. φHash's output width sets the classical search space; the post-Grover (quantum) figure is half the classical bit count, the standard quadratic Grover speedup, computed live from the actual digest length above. These are (second-)preimage numbers: classical n, quantum n/2. The obvious objection is the birthday bound — plain collision resistance is only n/2 classical, so shouldn't that be the real limit? For φSign it is not, and this is by construction. φSign is a stateful hash-based signature — a hypertree in the SPHINCS+ style (WOTS-type chains, FORS, Merkle trees; its own construction, not an SP 800-208 parameter set) that selects its one-time key from a monotonic counter, not at random. Two things follow. First, every hash call is tweaked by its position (layer, chain index, step), so an attacker never gets to run an offline, un-targeted birthday search — the relevant property is target-collision resistance (equivalently second-preimage): find a collision for a target fixed after the tweak, which costs n classical and n/2 quantum, not n/2 and n/3. Avoiding the birthday penalty is exactly why WOTS/XMSS/SPHINCS+ tweak their hashes. Second, because the counter picks the leaf, no one-time key is ever reused (the counter must be persisted; reuse is catastrophic, and FORS sits underneath as a defence-in-depth few-time net if it ever desyncs). So the table's figures are the operative ones for φSign: tier 0 is 232 classical / 116 quantum, tier 2 is 608 / 304. (Plain, un-tweaked collision resistance would matter to a bare hash used as a raw digest — there tier 0's 232-bit output is a 116-bit birthday target — but φSign is not used that way.) One caution on NIST categories: they are defined by the full gate-level cost of key search on AES or collision search on SHA, not by search exponents alone, so placing tier 0 at Category 3 and tier 2 past Category 5's 232-bit quantum floor is an exponent comparison, not a certified category claim.
φKEM (φ-KEM-304). Module-LWE, ring degree n=512, module rank k=3, q=3329, η=3 (CBD noise). Why n=512, k=3 and not n=256, k=6 (same 1,536 dimension, Kyber's proven geometry)? Because the RM(1,3) reconciliation forces it. RM(1,3) runs at rate 0.5 bits per coefficient (it spends 8 coefficients to carry 4 message bits), so a 256-bit shared secret needs 512 coefficients, and the ciphertext's single message polynomial must therefore be degree 512 → n=512. At n=256 that polynomial would carry only 128 bits, forcing a second message polynomial and a different ciphertext shape. So n=512 is downstream of the reconciliation code (chosen for CCA-soundness, below), not an arbitrary "make it bigger." The NTT cost of that choice is real and stated: q−1=3328=2⁸·13 gives only a primitive-256th root of unity, so at n=512 a full negacyclic NTT is impossible; the transform is hybrid/incomplete — 128 residue rings of degree 4, one split-level shallower than Kyber's 128 degree-2 leaves. n=512 is strictly worse for the NTT than n=256 would be; it is the price of carrying the whole secret in one RM(1,3)-coded polynomial. On the estimate: classical core-SVP β≈1462 against a lattice dimension of ≈1536. This is worth stating sharply: when β approaches the full dimension, BKZ-β is essentially a single near-full-dimension SVP call, and you have left the regime the core-SVP heuristic was fit in — so ≈427-bit classical and ≈387-bit quantum (the 0.265β constant) are best read as order-of-magnitude lower bounds, not tight levels. Even the Kyber team treats 0.265β as pessimistic (it assumes implausible quantum RAM). And core-SVP is a conservative lower-bound model, not a gate-level cost model, while NIST categories are defined by the full cost of attacking AES — so the precise claim is that both figures clear the suite's 304-bit floor and Category 5's 232-bit quantum bar under the core-SVP model, not that a certified gate-count comparison has been made. Read together, the numbers point one way: 157 bits of DFR margin, 2× ML-KEM-1024's wire size, β near full dimension — the parameters are conservative rather than tightly optimized, and there is no published cost function trading DFR against size against security the way Kyber's are. On the Reed–Muller reconciliation: RM(1,3) is the one φ-native load-bearing element inside encode/decode, and changing it changes more than hardness — it changes the decryption failure rate (DFR) that the FO transform's CCA2 argument depends on directly. The spec's own bound is δ ≤ 2⁻¹²⁸; RM(1,3) computes to δ = 2⁻²⁸⁵ against a no-reconciliation baseline of δ = 2⁻⁶⁹ that fails the bound outright — "computes to" and not "measures to": this is a closed-form convolution of per-coefficient error distributions, exact under that model, not sampled. The team's own review notes list the open question plainly: "is δ ≤ 2⁻¹²⁸ correct under a worst-case, not average-case, noise model?" — unresolved, not glossed over. The CCA2-relevant cost is also finding one failing ciphertext by directed search, not the bulk failure probability (failure boosting); with a 157-bit margin over the 2⁻¹²⁸ bound that's very likely still fine, but it hasn't been separately analyzed here. The code is load-bearing for CCA2 soundness, not merely a rate optimization, and this is exactly where a Kyber-derived scheme stops being Kyber.
φDSA. Fiat–Shamir with aborts (the ML-DSA algorithm) over the suite's shared proof-system field (N=256, Q=1342832641, k=l=9, η=7, τ=60) — one field across the proof system and the signature, so signatures can be verified inside recursive proofs natively. This uses SHAKE-256 throughout, not φHash — φHash's output isn't uniform enough for the rejection-sampling step (measured: ~6% of mask coefficients pegged at the bound vs. ~0.02% expected under a uniform XOF; see "The φHash-XOF finding, and the fix" above — the same defect, now fixed for φKEM's own use of φHash, but φDSA's rejection sampling has zero tolerance for any bias, so SHAKE-256 remains the correct choice regardless), so φDSA's hardness has zero φHash dependency, unlike every other primitive on this page. Both lattice problems, not just forgery. The estimator computes two attacks and the design balances them: MLWE key-recovery at 282.4 classical / 256.3 quantum, and SIS strong-unforgeability at 280.6 / 254.7. η=7 was chosen as the smallest noise that lifts MLWE key-recovery above the SIS ceiling, so forgery (≈255 quantum) is the binding constraint by design, not an unexamined one — the concern that a large modulus (Q≈2³⁰) makes key-recovery the weak axis is exactly what η=7 answers. Both clear ML-DSA-87 on the same model. That means dominance on every security axis, not size: φDSA's keys and signatures are larger (4,352 / 6,206 B vs 2,592 / 4,627 B — see the sizes table), a deliberate cost of the shared proof-system field. The estimator is checkable: it is the in-repo core-SVP/BKZ script (`test_dsa_derivation.cpp`), anchored by reproducing the published ML-DSA-2/5 MLWE core-SVP figures (123 / 252) before it is trusted on φDSA; the forgery collision sits in the norm-below-q regime where the conservative SIS bound is provable rather than asserted. Not byte-conformant FIPS-204 (own encodings, self-consistent against its own KATs — this proves the implementation matches this team's own spec, not that the spec itself has been independently reviewed; it is not ACVP validation).
Verify-in-proof. The shared field lets signature verification run inside recursive proofs — that is this page's first tab, where the proofs are verified live. The security-relevant detail: SHAKE-based Fiat–Shamir is not arithmetizable at any size, so the in-circuit path derives its challenge with a field-native Poseidon-class x⁷ sponge (width 3, 3×3 Cauchy MDS, 52 rounds — an algebraic-degree figure only) while ordinary signing keeps SHAKE-256. With a single capacity element over a ~2³⁰ field that sponge's generic security is capacity-limited (~2¹⁵ collision, ~2³⁰ preimage), so the first tab is scoped as a mechanism demonstration; a parameter revision deriving the capacity from the 128-bit target is in progress. The in-circuit statement verifies a different signature — the algebraic-FS variant — resting on an experimental, unreviewed hash from a family (Poseidon/Rescue/Griffin/Anemoi) with an active cryptanalytic record. It is the least battle-tested element on this page and is scoped as such in the first tab.
measured, not asserted

Sizes and measured costs

Wire sizes next to the corresponding FIPS 203/204/205 parameter sets, and native single-core timings from the suite's own benchmark harnesses (—). The φ-native instantiation has a real, measured price: φHash costs about 0.35 µs/byte against SHAKE-256's 0.009 µs/byte (≈38× on 176-byte inputs), and on the full KEM task the φHash path runs 3992/2468/1997 µs (keygen/encaps/decaps) against 356/401/479 µs for the same lattice core over SHAKE. These φHash figures are post-cap — the XOF-bias fix reduced the block to chandas=3, which costs ~11× more φHash calls per request (keygen went from ~355 µs to ~3992 µs), so this is the current, slower reality, not a pre-fix number. And note the direction this points: the KEM core is hash-agnostic, a NIST-validated SHAKE-256 instantiation exists and is 4–11× faster, and the one φHash defect found so far was found in exactly this path — so the honest reading is that SHAKE-256 is the lower-risk default and φHash the φ-native opt-in, a call the suite still has open. That cost buys the suite's own hash leg, not extra lattice security — stated here so nobody has to reverse-engineer it from the demo's speed.
